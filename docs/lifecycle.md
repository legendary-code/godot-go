# Object lifecycle

**Short answer:** for `Node` / `Node2D` / scene-tree types, the engine
manages lifetime — `queue_free()` or scene-tree removal disposes them
the same way it would in GDScript. For `Resource` / `Image` / any
`RefCounted`-derived class, the framework attaches a Go finalizer that
drops the engine reference automatically when Go's GC determines you
no longer hold the wrapper. You don't have to call `Free` or
`Unreference` in idiomatic code — drop the Go reference and the
engine-side cleanup happens for you, eventually.

## Two kinds of objects

Godot's class tree splits at `Object` vs `RefCounted`:

- **Plain `Object`** (and its non-RefCounted descendants — `Node`,
  `Node2D`, `Window`, etc.) is host-managed. The scene tree owns the
  lifetime. `queue_free()` schedules deletion; removing from the tree
  with `remove_child(...)` orphans an instance you then `free()`
  manually. Same model as GDScript.
- **`RefCounted`** (and everything derived from it — `Resource`,
  `Image`, `Texture2D`, `Shader`, `Animation`, `RegEx`, etc.) is
  reference-counted. Each holder bumps the count on acquire,
  decrements on release; the engine destructs at zero.

About 64% of engine classes (652 of 1023) are RefCounted-derived —
every asset type and most non-node helpers. The framework treats them
differently from plain Objects automatically.

## What the framework does for you

Every RefCounted-derived class has a generated `XFromPtr` helper that
attaches a Go finalizer:

```go
// Generated by bindgen for every RefCounted class:
func ResourceFromPtr(p gdextension.ObjectPtr) *Resource {
    if p == nil { return nil }
    ret := &Resource{}
    ret.BindPtr(p)
    runtime.SetFinalizer(ret, func(r *Resource) {
        gdextension.RunOnMain(func() { r.Unreference() })
    })
    return ret
}
```

When Go's garbage collector marks the `*Resource` wrapper as
unreachable, the finalizer runs (on a runtime-internal goroutine,
not the main thread). The framework bounces the `Unreference` call
through `gdextension.RunOnMain` so it lands on the engine's main
thread regardless. The engine's refcount drops by one; if that takes
the count to zero, the engine destructs the Resource.

For users this means: **drop the Go reference and the cleanup happens
automatically**. No explicit `Free` calls, no `defer r.Unreference()`
boilerplate, no leaked resources from forgotten cleanup.

## Drain timing matters

The finalizer's `RunOnMain` enqueues onto the framework's main-thread
work queue. Until something *drains* the queue, the engine
`Unreference` doesn't actually run. The standard recipe (see
[`docs/threading.md`](./threading.md)) is to call
`runtime.DrainMain()` once per frame from your `Process` override:

```go
// @override
func (n *MyNode) Process(delta float64) {
    runtime.DrainMain()
    // ... per-frame logic ...
}
```

A typical refcount lifecycle then looks like:

| Step                              | Refcount | Notes                              |
| --------------------------------- | -------- | ---------------------------------- |
| `r := core.RegExCreateFromString` | 1        | engine returns +1 owned reference  |
| Go reference dropped (`r = nil`)  | 1        | wrapper unreachable; awaiting GC   |
| `runtime.GC()` (or natural GC)    | 1        | Go GC marks; finalizer queued      |
| `runtime.DrainMain()` next frame  | 0        | engine destructs the RegEx         |

The destruction lands on the next frame after Go's GC notices.
Typical-case latency: a couple frames. Game-state-critical resources
that need deterministic free can call `r.Unreference()` directly —
the finalizer is harmless after refcount hits zero.

If your extension has no per-frame `Process` (editor plugins,
singleton-only extensions), drain wherever you have a main-thread
hook — a tool method invoked from GDScript, an editor button
callback, etc.

## When you'd reach for explicit `Unreference`

The finalizer model is the right default. There are a few cases
where you'd want explicit ref/unref:

- **Deterministic free under tight memory budgets.** A 4K texture
  held one frame too long can blow VRAM. `tex.Unreference()` right
  after you're done is the answer.
- **Holding extra references on purpose.** If you stash a
  `*Resource` in two long-lived structs, you want both to keep it
  alive even if the original Go reference goes away. `r.Reference()`
  on each storage site does that; pair with `Unreference()` on
  release.
- **Bridging to non-Go ownership.** If you're handing a raw
  `ObjectPtr` to a C library that holds onto it, you need to keep
  the engine count up manually until that library drops it.

Calling `Unreference` after the finalizer has already run is safe —
the engine refcount is already at zero, the destruction has already
happened, and the call is a no-op against a freed object… *if* you
don't dereference the wrapper afterward. The framework doesn't
prevent that; it's user code's responsibility not to keep using a
wrapper after explicit `Unreference()`.

## What gets a finalizer

Every class with `is_refcounted: true` in `extension_api.json` gets
the finalizer treatment. As of Godot 4.6.2 that's:

- `RefCounted` itself
- `Resource` and every `Resource`-derived class (`Image`,
  `Texture2D`, `Animation`, `Shader`, ...)
- Standalone refcounted helpers (`RegEx`, `RegExMatch`, `JSON`,
  `XMLParser`, ...)
- All `Ref<T>` payloads in method signatures and properties

Plain `Object` subclasses (`Node`, `Node2D`, `Window`,
`EditorPlugin`, ...) keep the simpler `XFromPtr` — no finalizer,
because the host owns their lifetime.

## Singletons aren't refcounted

Godot's singletons (`Engine`, `OS`, `Input`, `ResourceLoader`, ...)
all derive from plain `Object`, not `RefCounted`. The host owns the
singleton instances for the engine's full lifetime; the framework
caches the wrapper via `sync.OnceValue` and never touches the
refcount. There's no leak risk and no finalizer attached.

## Edge cases worth knowing

- **Process exit.** Go's GC isn't guaranteed to run before
  `os.Exit`. On Windows, `internal/gdextension/shutdown.go` calls
  `os.Exit(0)` from Core deinit (see
  [`docs/hot-reload.md`](./hot-reload.md) for why). Any pending
  finalizers don't run. The OS reclaims memory; the engine doesn't
  get to log "leaked X resources" because the process already
  terminated. Acceptable trade-off.
- **Goroutine resurrection.** If a finalizer body somehow ends up
  resurrecting the object (storing the wrapper somewhere reachable
  again), Go won't call the finalizer again on subsequent
  unreachability. The framework's finalizer is a clean
  `Unreference()` call; no resurrection path. User code that
  composes additional finalizer logic should be careful.
- **Reference loops between Go and engine.** A user struct that
  holds a `*Resource` AND is itself owned by the Resource (somehow,
  via a callback registration) would form a Go GC cycle. Go's GC
  handles cycles, but the *engine* doesn't — if the resource holds
  a strong reference to your Go-side closure, neither side ever
  drops. Avoid setting up bidirectional ownership; let the
  shorter-lived side hold the reference.
